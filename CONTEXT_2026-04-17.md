## 1. SISTEMA — O QUE É

**SUS Raio-X** — sistema GovTech para gestão de filas de exames de Raio-X / consultas do SUS em **Montes Claros/MG**. MVP para apresentação em banca (~20 dias). Stack: React 18 + Vite + Tailwind. Backend: Supabase (PostgreSQL + Realtime + Auth + RPC SECURITY DEFINER).

**Problema resolvido:** vagas de exame canceladas em cima da hora ficavam ociosas. O sistema convoca automaticamente o próximo paciente da fila (FIFO clínico) quando há cancelamento — eliminando desperdício e garantindo equidade de acesso.

## 3. BANCO DE DADOS — SCHEMA CRÍTICO

Sempre é bom você perguntar qual o schema atual e pedir algumas queryes para confirmar o schema antes de mudanças em RPC ou qualquer coisa que relacione efetivamente com o banco.

### Tabelas principais
| Tabela | Pontos críticos |
|---|---|
| `patients` | `id, cns, nome, telefone` — seed gerou duplicatas por nome, SQL de limpeza criado |
| `queue_entries` | `patient_id, ubs_id, tipo_atendimento, nome_grupo_procedimento, status_local, prioridade_codigo` — `nome_grupo_procedimento` nullable |
| `appointments` | **NÃO tem `patient_id` direto** — vai via `queue_entry_id → queue_entries.patient_id`. Campos: `status` (enum: agendado/confirmado/cancelado), `st_paciente_avisado` (smallint, pode ser NULL), `equipment_id` |
| `notification_log` | `patient_id, appointment_id, tipo, canal, mensagem, resposta_paciente` (enum: confirmou/cancelou), `enviado_at, respondido_at, entregue` |
| `equipment` | `ubs_id, nome, tipo_atendimento, turno`, sem patient_id |
| `ubs` | `id, nome, tipo` — UBS reguladoras têm `tipo='R'` (bug: código antigo usava `'S','A'`) |

### View
`v_dashboard_fila` — join de queue_entries + patients + ubs + appointments + equipment

### RPC ATIVA: `executar_reaproveitamento` (v3)
Critérios FIFO (em ordem):
1. Mesma `ubs_id`
2. Mesmo `tipo_atendimento`
3. Mesmo `nome_grupo_procedimento`
4. Menor `prioridade_codigo` (menor = mais urgente)
5. Menor `data_solicitacao_sisreg` (mais antigo na fila)

**Guarda de segurança:** se `nome_grupo_procedimento` for NULL no appointment cancelado, encerra sem reaproveitamento.

**Frontend wrapper** (`orquestracao.js`):
```js
const { nomeConvocado, erro, nivelFallback } = await executarReaproveitamento(appointment_id)
// nivelFallback: 0=strict(prod), 1=sem proc, -1=fila vazia/erro
```
Retorna `result.nomeConvocado` (string|null). `nivel_fallback` não é retornado pela v3 — defaults a 0 (sem crash).

**v4 (rascunho local, NÃO aplicado):** adiciona fallback UBS+tipo sem proc. Tem PASSO 2C cross-UBS que o CTO rejeitou. Antes de aplicar: remover PASSO 2C do arquivo.

### Lógica de fetchContacts
Dois queries paralelos: `notification_log` + `appointments` (agendado/confirmado, futuros).
Merge: 1 item por `patient_id`. Contatos sem `notification_log` recebem `resposta_paciente: 'sem_notificacao'` (sentinel).
Sort: `resposta_paciente === null` primeiro (pendentes reais), depois por `enviado_at DESC`.

### `pendingNotif`
```js
const pendingNotif = [...chatNotifs].reverse().find(n =>
  !n.resposta_paciente &&
  (!n.scheduled_at || new Date(n.scheduled_at) > new Date())
) ?? null
```
Só exibe botões se agendamento ainda é futuro. Pacientes com histórico passado não recebem botões.

### Handlers
- `handleConfirmar(notif)` → update `notification_log` + `appointments status='confirmado'` → `setResultadoOrquestracao({ tipo: 'confirmou', nomeConvocado: null })`
- `handleCancelarConfirmado(notif)` → update log + `status='cancelado'` → `executarReaproveitamento` → `setResultadoOrquestracao({ tipo: 'cancelou', nomeConvocado })`
- `handleSelectContact(patientId)` → `setSelectedPatientId`
- Ao mudar contato: `setResultadoOrquestracao(null)` + `fetchChat()`

### Chat central (coluna 2)
- Histórico de `chatNotifs`: `SysBubble` (mensagem do sistema) + `PatientBubble` (resposta)
- Após `chatNotifs`, antes do `chatEndRef`: nota de orquestração visível só para operador:
```jsx

  <div className="flex justify-center my-3">
    <div className={`border rounded-xl px-4 py-2 text-xs text-center max-w-sm ...`}>
      {nomeConvocado
        ? `✅ Vaga reaproveitada — ${nomeConvocado} convocado da fila`

```
- Footer: só label `"Visão do Operador — mensagens enviadas automaticamente pelo sistema"`. **Sem input, sem botões de template**.

Usa `useNotifications` hook → `stats.total_hoje`, `stats.taxa_confirmacao`, `stats.sem_resposta`.

### Seção 2: Vagas sem confirmação (fetchVagas)
Query corrigida:
```js
.eq('status', 'agendado')
.or('st_paciente_avisado.is.null,st_paciente_avisado.eq.0')  // fix: NULL ≠ 0 no Postgres
.gte('scheduled_at', new Date().toISOString())
.lte('scheduled_at', limite)  // horasConfig padrão = 48h
```
**Otimista:** `setVagasEmRisco(prev => prev.filter(v => v.id !== vaga.id))` ao clicar "Notificar" — remove imediatamente, rollback via `fetchVagas()` se insert falhar.

### Notificar individual (`notificarVaga`)
1. Remove otimisticamente da lista
2. INSERT em `notification_log` (tipo: `lembrete_manual`, data_source: `manual`)
3. Se erro → rollback, toast de erro
4. Se ok → `appointments.update({ st_paciente_avisado: 1 })`
5. `refresh()` para atualizar histórico

### Notificar em massa (`notificarMassa`)
- Fecha modal + zera lista imediatamente (optimistic)
- Salva `vagasSalvas = [...vagasEmRisco]` para rollback
- INSERT batch em `notification_log`
- Se erro → `fetchVagas()` (rollback), se ok → `appointments.update` em batch

### Simular resposta (`simularResposta`)
- `'confirmou'` → update log + `appointments.status = 'confirmado'`
- `'cancelou'` → update log + `appointments.status = 'cancelado'` + `executarReaproveitamento(appointment_id)` → Toast especial com nome convocado

### Badge "Vaga Recuperada"
Aparece no histórico quando `n.resposta_paciente === 'cancelou'`.
