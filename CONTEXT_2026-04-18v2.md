## 3. Depuração do Fluxo Board 1 → Board 2

### Problema Relatado
Ao clicar "Notificar" em um paciente do Board 1, ocorria um **flicker** breve no Board 2 seguido de retorno ao Board 1. O paciente nunca persistia no Board 2.

### Anatomia do Bug (Raiz Composta — 3 iterações de diagnóstico)

#### Iteração 1 — `respondedPatientIds` por `patient_id`
O `respondedPatientIds` usava `patient_id` (a pessoa) como chave de exclusão. Qualquer paciente com histórico de confirmação/cancelamento quedava permanentemente banido do Board 2, mesmo com nova notificação pendente.

**Fix**: removido `respondedPatientIds`.

#### Iteração 2 — `naoNotificadosIds` por `patient_id`
O safety-net `naoNotificadosIds` usava `patient_id`. Um paciente com appointment B pendente (Board 1) e appointment A notificado (Board 2) ficava bloqueado do Board 2 pelo appointment B.

**Fix**: migrado para `naoNotificadosApptIds` com chave `appointment_id`.

#### Iteração 3 (causa raiz definitiva) — Race condition Realtime vs. UPDATE
**O fluxo de `notificarVaga` é:**
1. INSERT em `notification_log` para appointment AX
2. Realtime (`notification_log`) dispara → `fetchNotifications()` → `notifications` atualiza → AX aparece em Board 2 (**flicker**)
3. UPDATE `appointments.st_paciente_avisado = 1` em AX

**Entre os passos 2 e 3**, o Realtime em `appointments` pode disparar `fetchVagas()`. Se esta query chega ao banco antes do UPDATE do passo 3 commitar (latência de rede/propagação), `vagasEmRisco` recebe AX de volta com `st_paciente_avisado = null`. Com `naoNotificadosApptIds.has(AX.id) = true`, a notificação de AX é bloqueada do Board 2. O paciente volta ao Board 1.

### Solução Definitiva

**Fonte de verdade:** `notification_log` — não `st_paciente_avisado`.

```javascript
// Set de appointment_ids que já têm notification_log (notificados = Board 2 ou 3)
const notifiedApptIds = useMemo(
  () => new Set(notifications.map(n => n.appointment_id).filter(Boolean)),
  [notifications]
)

// Board 1: filtra primeiros por notifiedApptIds, depois deduplica por paciente
const naoNotificados = useMemo(() => {
  const seen = new Set()
  return vagasEmRisco.filter(v => {
    if (notifiedApptIds.has(v.id)) return false  // já notificado → Board 2 ou 3
    const pid = v.paciente_id
    if (pid && seen.has(pid)) return false        // dedup seed
    if (pid) seen.add(pid)
    return true
  })
}, [vagasEmRisco, notifiedApptIds])
```

**Por que funciona:** O INSERT em `notification_log` é o primeiro passo de `notificarVaga`. A Realtime de `notification_log` dispara `fetchNotifications()` e imediatamente o `notifiedApptIds` passa a conter `AX.id`. Mesmo que `fetchVagas()` traga AX de volta (por race condition com o UPDATE de `st_paciente_avisado`), `naoNotificados` já o filtra por `notifiedApptIds`. A transição Board 1 → Board 2 é determinística e atômica no momento do INSERT.

O campo `st_paciente_avisado` continua sendo marcado por integridade (fallback para filtro de longo prazo) mas não é mais a fonte de verdade para o frontend.

## 5. Fluxo de Dados por Board

### Board 1 → Board 2 (via botão "Notificar")
```
notificarVaga(vaga) {
  1. INSERT notification_log (appointment_id = vaga.id) ← gate de transição
  2. Realtime → fetchNotifications() → notifiedApptIds inclui vaga.id
  3. naoNotificados exclui vaga (notifiedApptIds) → Board 1 perde o paciente
  4. aguardandoResposta inclui a notificação → Board 2 ganha o paciente
  5. UPDATE appointments.st_paciente_avisado = 1 (integridade, não crítico para UI)
  6. await Promise.all([refresh(), fetchVagas()])
}
```

### Board 2 → Board 3 (via Simular ou WhatsApp real)
```
simularResposta(notif, 'confirmou'|'cancelou') {
  1. UPDATE notification_log.resposta_paciente
  2. UPDATE appointments.status = 'confirmado'|'cancelado'
  3. Se 'cancelou': executarReaproveitamento() (RPC FIFO)
  4. await Promise.all([refresh(), fetchVagas()])
}
```

### Board 2 → Board 1 (Renotificar)
```
renotificarPaciente(notif) {
  → chama notificarVaga com shape adaptado do notification_log para vaga
  → cria nova entrada em notification_log
  → paciente permanece no Board 2 (nova notificação mais recente)
}
```

## 7. Pendências Conhecidas

### Frontend
- [ ] `notifiedApptIds` usa os 100 registros mais recentes de `notification_log`. Em produção com histórico grande, appointments antigos (fora do limit=100) poderiam reaparecer incorretamente no Board 1.
