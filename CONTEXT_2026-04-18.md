## 1. SISTEMA — RESUMO EXECUTIVO

**SUS Raio-X** — plataforma GovTech/HealthTech para gestão inteligente de filas de exames e consultas do SUS em Montes Claros/MG. MVP para apresentação em banca. Stack: React 18 + Vite + Tailwind. Backend: Supabase (PostgreSQL + Realtime + Auth + Edge Functions via RPC SECURITY DEFINER).

**Problema resolvido:** vagas de exames canceladas a poucos dias/horas do procedimento ficavam ociosas no SUS. O sistema: (1) envia notificações WhatsApp aos pacientes agendados para confirmar presença, (2) detecta cancelamentos e aciona automaticamente o próximo paciente elegível da fila de espera (FIFO clínico), eliminando desperdício e garantindo equidade de acesso.

## 3. BANCO DE DADOS — SCHEMA CRÍTICO

> **Regra de ouro:** sempre confirmar schema com query antes de qualquer mudança em RPC ou migration. Solicitar resultado de `information_schema.columns WHERE table_schema='public'` antes de propor alterações.

### Tabelas principais

| Tabela | Pontos críticos |
|---|---|
| `patients` | `id, cns, nome, telefone` — seed gerou duplicatas; SQL diagnóstico/limpeza em `202604170001` |
| `queue_entries` | `patient_id, ubs_id, tipo_atendimento (ENUM), nome_grupo_procedimento (text, nullable), status_local (ENUM), prioridade_codigo (smallint)` |
| `appointments` | **SEM `patient_id` direto** — acesso via `queue_entry_id → queue_entries.patient_id`. `status` ENUM: agendado/confirmado/cancelado. `st_paciente_avisado` (smallint, nullable). `reaproveitado_de_id` (uuid, rastreabilidade FIFO) |
| `notification_log` | `patient_id, appointment_id, tipo (ENUM), canal, mensagem, resposta_paciente (ENUM: confirmou/cancelou), enviado_at, respondido_at, entregue` |
| `equipment` | `ubs_id, nome, tipo_atendimento (ENUM), turno (ENUM)` |
| `ubs` | UBS reguladoras: `tipo = 'R'` (bug histórico: código antigo usava `'S','A'`) |

### ENUMs críticos (USER-DEFINED — não aceitam cast implícito de text em PL/pgSQL)

```sql
tipo_atendimento   -- 'exame' | 'consulta' | ...
status_local       -- 'aguardando' | 'agendado' | 'cancelado' | ...
data_source        -- 'manual' | 'seed_demo' | 'sisreg' | ...
```

> **Lição aprendida:** em PL/pgSQL, variáveis `text` NÃO são casting implicitamente para ENUMs.
> Usar `INSERT...SELECT` puxando o valor diretamente de uma row existente, ou usar `::nome_do_enum`.
> Este bug causou falha no BLOCO C do seed (`tipo_atendimento text := 'exame'` → erro 42804).
> Correção: substituir `VALUES (... v_tipo ...)` por `SELECT ... qe.tipo_atendimento ...` de uma row real.

### View
`v_dashboard_fila` — join de queue_entries + patients + ubs + appointments + equipment

## 4. RPC ATIVA: `executar_reaproveitamento` (v3)

**Arquivo:** `202604150003_fix_rpc_fifo_criteria.sql`

**Critérios FIFO clínico (ordem obrigatória):**
1. `status_local = 'aguardando'`
2. Mesma `ubs_id`
3. Mesmo `tipo_atendimento`
4. Mesmo `nome_grupo_procedimento`
5. `prioridade_codigo ASC` (1 = urgente, 2 = rotina)
6. `data_solicitacao_sisreg ASC` (mais antigo = mais tempo na fila)

**Comportamento:**
- PASSO 1: marca o paciente cancelado com `status_local = 'cancelado'`
- PASSO 2: busca próximo elegível (exclusão explícita por queue_entry_id E patient_id)
- PASSO 3: promove convocado para `status_local = 'agendado'`
- PASSO 4: cria novo appointment com `reaproveitado_de_id` preenchido (rastreabilidade)
- Retorna: `{ nomeConvocado: string|null, erro: string|null }`
- Se `nome_grupo_procedimento` é NULL: encerra sem reaproveitamento (guarda de segurança)
- SECURITY DEFINER — executado com permissão de service_role, não de user

**Wrapper frontend** (`orquestracao.js`):
```js
const { nomeConvocado, erro, nivelFallback } = await executarReaproveitamento(appointment_id)
// nivelFallback: 0=strict, 1=sem proc, -1=fila vazia/erro (v3 retorna 0 por padrão)
```

**v4 (NÃO aplicada):** `202604160001_fix_rpc_null_proc_fallback.sql`
- Adiciona fallback em cascata: strict → sem proc → (cross-UBS REJEITADO)
- Antes de aplicar: **remover bloco PASSO 2C** (cross-UBS rejeitado pelo CTO — sem base clínica)

## 5. MUDANÇAS DESTA SESSÃO

### 5.1 Bug Root Cause Analysis: "Paciente notificado/cancelado reaparece na fila"

**Sintoma observado:**
- Márcio Cunha Barbosa foi notificado → apareceu no WhatsApp com histórico correto
- Márcio cancelou via Simular Resposta → badge "Vaga Recuperada" apareceu no Histórico
- **Mas:** ao voltar para Notificações, Márcio still aparecia no "Vagas sem confirmação"

**Diagnóstico técnico — 3 bugs independentes que se somavam:**

**BUG 1 — `vagasFiltradas` não excluía cancelamentos** (gravidade: CRÍTICA)
```js
// ANTES — filtra apenas 'confirmou', ignora 'cancelou'
notifications.filter(n => n.resposta_paciente === 'confirmou')
```
`notification_log.resposta_paciente = 'cancelou'` era gravado corretamente (visível no Histórico),
mas a lógica de exclusão da lista de "Vagas" verificava SÓ `'confirmou'`. Resultado: qualquer
paciente que cancelou continuava elegível para aparecer na lista.

**BUG 2 — `notificarVaga` não tinha mark direto por `appointment.id`** (gravidade: MÉDIA)
O código ia buscar todos os `queue_entries` do paciente para marcar via bulk. Se esse
join retornasse array vazio (possível em dados inconsistentes ou via RLS), nenhum UPDATE
era executado e `st_paciente_avisado` permanecia `NULL`, causando reexibição na próxima
`fetchVagas()`.

**BUG 3 — `notificarVaga` e `notificarMassa` não chamavam `fetchVagas()` no final** (gravidade: MÉDIA)
Ambas as funções chamavam apenas `refresh()` (que atualiza `useNotifications` / Histórico),
mas NÃO chamavam `fetchVagas()` para resincronizar a lista de Vagas com o banco. A lista
ficava dessincronizada até o próximo evento Realtime de `appointments`.

### 5.2 Correções aplicadas em `NotificacoesPage.jsx`

**Fix 1 — `respondedPatientIds` inclui ambos os estados de resposta:**
```js
// DEPOIS — exclui quem confirmou OU cancelou
notifications.filter(n => n.resposta_paciente === 'confirmou' || n.resposta_paciente === 'cancelou')
```

**Fix 2 — `notificarVaga`: mark direto antes do bulk-mark:**
```js
// Garantia primária: marca o appointment específico diretamente
await supabase.from('appointments')
  .update({ st_paciente_avisado: 1 })
  .eq('id', vaga.id)
// Cobertura secundária: marca todos os appointments deste paciente (multi-appointment seed)
```

**Fix 3 — `notificarVaga` e `notificarMassa`: fetchVagas no final:**
```js
// DEPOIS
await Promise.all([refresh(), fetchVagas()])
```

### 5.3 Fix anterior: BLOCO C do seed (`202604170002`)

**Erro:** `ERROR 42804: column "tipo_atendimento" is of type tipo_atendimento but expression is of type text`

**Causa:** PL/pgSQL não faz cast implícito de `text` para ENUM USER-DEFINED.
```sql
-- ANTES (falha em runtime)
v_tipo text := 'exame';
INSERT INTO queue_entries (..., tipo_atendimento, ...) VALUES (..., v_tipo, ...)

-- DEPOIS (correto)
INSERT INTO queue_entries (..., tipo_atendimento, ...)
SELECT ..., qe.tipo_atendimento, ...
FROM appointments a JOIN queue_entries qe ON qe.id = a.queue_entry_id
WHERE a.status = 'agendado' AND a.scheduled_at > now() AND qe.ubs_id = v_ubs_id
LIMIT 1;
```

### 6.2 Integração WhatsApp

**MVP (atual):**
- Simulação completa: operador clica "Simular" para emular resposta do paciente
- Mensagens são geradas localmente e salvas em `notification_log` (sem envio real)
- Templates institucionais (`gerarMensagemSistema`, `gerarTemplateConfirmacao`, `gerarTemplateReuso`) armazenados apenas em memória do frontend

**Produção — requisitos para WhatsApp Business API:**
```
1. APROVAÇÃO META: conta WhatsApp Business API aprovada
   → requires: CNPJ da prefeitura, documento oficial, número dedicado

   → Idempotência: verificar messageId antes de processar (evitar duplicatas)

4. EDGE FUNCTION SUPABASE:
   → Mover lógica de webhook para supabase/functions/whatsapp-webhook/

   → Tabela notification_log atualizada atomicamente com resposta + appointment status

5. TEMPLATES APROVADOS PELA META:
   → Templates de notificação precisam ser aprovados (1-3 dias úteis)
   → Parâmetros variáveis (nome, data) devem seguir formato {{1}}, {{2}}
   → Separar: templates de CONFIRMAÇÃO, templates de CANCELAMENTO, templates de CONVOCAÇÃO

6. CONFORMIDADE LGPD:
   → Consentimento explícito do paciente para receber WhatsApp (capturado no cadastro)
   → Campo consent_whatsapp + consent_at em patients
   → Opt-out deve ser processado imediatamente (STOP → bloquear envios futuros)
   → Retenção de logs: política de retenção de notification_log (sugestão: 2 anos para auditoria SUS)
```

### 6.3 Orquestração FIFO e integridade clínica

**MVP (atual):**
- RPC v3 com critérios: UBS + tipo_atendimento + nome_grupo_procedimento + prioridade + data
- Sem fallback: se não encontra candidato exato, retorna fila vazia
- Sem verificação de disponibilidade de equipamento no horário convocado
- Sem timeout para resposta do convocado

**Produção:**
```
1. RPC v4 (rascunho em 202604160001): fallback UBS+tipo sem proc (revisar antes de aplicar)
   → ATENÇÃO: remover PASSO 2C (cross-UBS foi REJEITADO pelo CTO)

2. Verificação de disponibilidade de equipamento:
   → Antes de criar novo appointment, verificar se equipment_id está disponível no horário
   → Evitar overbooking quando múltiplos cancelamentos simultâneos disparam RPCs em paralelo
   → Solução: SELECT...FOR UPDATE no equipment ou controle de concorrência por slot

3. Timeout de resposta do convocado:
   → Paciente convocado tem N horas para confirmar
   → Job scheduler (pg_cron ou Edge Function Cron) para liberar vaga não confirmada
   → Convocar próximo se timeout expirar sem confirmação

4. Auditoria completa:
   → Campo reaproveitado_de_id já presente em appointments (excelente para auditoria)
   → Log de cada decisão do FIFO: por que X foi convocado antes de Y
   → Relatório de reaproveitamento para Secretaria de Saúde

5. Prioridade clínica revisada:
   → prioridade_codigo atual: 1=urgente, 2=rotina
   → Produção: alinhar com classificação SISREG (pode ter mais categorias)
   → Revisar com equipe clínica se prioridade deve sobrepor antiguidade em todos os casos
```

## 11. DECISÕES ARQUITETURAIS CONSOLIDADAS

| Decisão | Resolução | Justificativa |
|---|---|---|
| Cross-UBS fallback no RPC | **REJEITADO** | Sem base clínica para priorizar UBS "mais próxima" |
| Double-check de cancelamento | **No PainelPaciente** (estado local) | Não polui o chat central |
| Linguagem no PainelPaciente | **Linguagem paciente** | "Sua vaga foi para o próximo da fila" |
| Linguagem no chat central | **Linguagem operador** | "✅ Vaga reaproveitada — X convocado da fila" |
| Footer chat central | **Sem input, sem botões** | Só label "Visão do Operador" |
| EventBubbles no chat | **Removidos** | Resultado de orquestração vai apenas ao PainelPaciente + nota center |
| Badge sem notification_log | `'sem_notificacao'` sentinel → badge cinza "Ag. Notif." | Evitar null checks em cascata |
| Remoção de notificados | **Optimistic UI** + rollback se erro | UX responsiva em demo |
| UBS reguladoras | `tipo = 'R'` | Bug corrigido de `.in(['S','A'])` |
| Mark de st_paciente_avisado | **Direto por ID** + bulk secundário | Garantia mesmo com dados inconsistentes |
| Exclusão de fila de vagas | **Por resposta (confirmou OU cancelou)** | Ambas respostas encerram o ciclo de notificação |
