## 1. SISTEMA — O QUE É

**SUS Raio-X** — sistema GovTech para gestão de filas de exames de imagem e consultas
especializadas do SUS em **Montes Claros/MG**.

**Edital:** CPSI Co.NE nº 004/2026 — Prefeitura de Montes Claros / ENAP / Sudene / BID
**Submissão MVP:** 28/04/2026 | **Aceleração:** 18/mai → 10/jul/2026 | **Demoday:** 03–07/ago/2026
**Valores:** R$200k (CPSI) → R$1,6M (contrato subsequente)

**Problema resolvido:** vagas de exame canceladas em cima da hora ficavam ociosas. O sistema:
1. Convoca automaticamente o próximo paciente da fila (FIFO clínico) ao haver cancelamento
2. Monitora ocupação de equipamentos em tempo real
3. Comunica com pacientes via WhatsApp
4. Oferece assistente de IA conversacional para o gestor público

**Stack:** React 18 + Vite + Tailwind | Supabase (PostgreSQL + Realtime + Auth + RPC SECURITY DEFINER)

**Metas do edital CPSI 004/2026:**
| KPI | Baseline | Meta |
|---|---|---|
| Absenteísmo | ~35% | ≤ 15% |
| Tempo médio espera | > 120 dias | ≤ 120 dias |
| Aproveitamento de capacidade | ~65% | ≥ 85% |
| Satisfação usuário | ? | > 8/10 |

### 5.2 Configuração técnica

| Item | Valor |
|---|---|
| Modelo | `claude-sonnet-4-20250514` |
| URL API (dev) | `/api/anthropic/v1/messages` (proxy Vite) |
| URL API (prod/TODO) | Supabase Edge Function |
| Proxy Vite | `changeOrigin: true` + `proxy.on('proxyReq', removeHeader('origin'))` |
| API Key | `VITE_ANTHROPIC_API_KEY` em `.env.local` (não commitar) |

**Por que o proxy remove o header `origin`:** o Anthropic detecta requests de browser
pelo header `Origin`. Com `changeOrigin: true` o Vite troca apenas o `Host`, mas passa
adiante o `Origin`. A remoção explícita via `proxyReq.removeHeader('origin')` faz o
Anthropic tratar a request como server-to-server, eliminando o erro CORS.

#### 2.1 `get_fila_por_ubs(p_horizonte_dias, p_tipo_atendimento)`
**Responde:** "Qual UBS tem maior fila?", "Onde está a demanda represada?"
```sql
-- queue_entries com status_local IN ('aguardando','agendado') → agrupa por ubs
-- Retorno: {ubs_nome, municipio, total_aguardando, pct_do_total, espera_media_dias}
SELECT
  u.nome AS ubs_nome,
  u.municipio,
  COUNT(qe.id) AS total_aguardando,
  ROUND(COUNT(qe.id)*100.0 / NULLIF(SUM(COUNT(qe.id)) OVER (), 0), 1) AS pct_do_total,
  AVG(EXTRACT(EPOCH FROM (NOW()-qe.data_solicitacao_sisreg))/86400)::int AS espera_media_dias
FROM queue_entries qe
JOIN ubs u ON u.id = qe.ubs_id
WHERE qe.status_local IN ('aguardando','agendado')
  AND ($1 IS NULL OR qe.data_solicitacao_sisreg >= NOW() - ($1||' days')::interval)
GROUP BY u.id, u.nome, u.municipio
ORDER BY total_aguardando DESC
```
**Novo intent:** `fila_por_ubs`

#### 2.4 `get_tipos_exame_solicitados(p_horizonte_dias, p_tipo_atendimento)`
**Responde:** "Quais exames são mais solicitados?", "Tipo de procedimento mais pedido?"
```sql
-- queue_entries.nome_grupo_procedimento (com fallback para tipo_atendimento)
-- Retorno: {tipo_exame, total_solicitacoes, pct_do_total, espera_media_dias}
SELECT
  COALESCE(qe.nome_grupo_procedimento, qe.tipo_atendimento, 'Não informado') AS tipo_exame,
  COUNT(*) AS total_solicitacoes,
  ROUND(COUNT(*)*100.0 / NULLIF(SUM(COUNT(*)) OVER(), 0), 1) AS pct_do_total,
  AVG(EXTRACT(EPOCH FROM (NOW()-qe.data_solicitacao_sisreg))/86400)::int AS espera_media_dias
FROM queue_entries qe
WHERE $1 IS NULL OR qe.data_solicitacao_sisreg >= NOW() - ($1||' days')::interval
GROUP BY tipo_exame
ORDER BY total_solicitacoes DESC LIMIT 15
```
**Novo intent:** `tipos_exame`

## 14. Sessão 2026-04-19 — Fix get_exames_por_local + Qualidade de código

### Problema diagnosticado
A intent `exames_por_local` no `AssistenteIA` retornava "não consegui identificar dados" mesmo com 286 registros `status='realizado'` no banco.

**Root cause (confirmado via `pg_get_functiondef`):**
A função `get_exames_por_local` existia em 2 overloads quebrados no Supabase:

| Overload | Problema |
|----------|---------|
| v1 `(integer)` | Usava `a.nome_unidade_executante` — coluna inexistente em `appointments` |
| v2 `(integer, text)` | Retornava `{nome, total}` — contrato incompatível com todos os callers |

O JS lia `e.equipamento_nome` → `undefined` → `Number(undefined) = NaN` → `NaN > 0 = false` → filter zerava todas as linhas → narrador recebia array vazio.

**Diagnóstico-chave:**
```sql
-- Confirma existência de dados:
SELECT status, count(*) FROM appointments GROUP BY status ORDER BY count(*) DESC;
-- → realizado=286, faltou=135

-- Expõe o contrato real da função:
SELECT pg_get_functiondef(oid) FROM pg_proc WHERE proname = 'get_exames_por_local';
-- → revelou retorno {nome, total} em vez de {equipamento_nome, unidade_nome, realizados, ...}
```

**Lição PostgreSQL:** `CREATE OR REPLACE FUNCTION` não pode alterar a lista de colunas de `RETURNS TABLE` → **ERROR 42P13**. É obrigatório `DROP FUNCTION` explícito antes de recriar.

### Migration 0009 aplicada
`supabase/migrations/202604190001_fix_rpc_exames_por_local.sql`

- DROP ambos overloads (v1 coluna inexistente + v2 contrato errado)
- CREATE com contrato canônico: `{equipamento_nome, unidade_nome, realizados, total_agendado, faltas, taxa_absenteismo}`
- Padrão LEFT JOIN: todos os equipamentos `status='ativo'` retornados, mesmo com `realizados=0`
- `taxa_absenteismo` alinhada com `calcular_absenteismo()`: denominador = `realizados + faltou` (não `total_agendado`)
- Bloco DO $$ valida: exatamente 1 overload + `get_exames_por_local(90, NULL)` retorna linhas

#### Erro 2: `P0001 Nenhum equipment ativo encontrado`
**Causa:** Após BLOCO 0 deletar todo o equipment seed, `SELECT FROM equipment WHERE status='ativo'`
retornava vazio. O bloco ENUM usava isso para copiar os tipos.

**Fix:** Mudado para `SELECT ... FROM equipment LIMIT 1` (qualquer registro, ativo ou não).
Adicionado fallback via `pg_enum` para quando a tabela está vazia.

#### Erro 3: `3F000 schema "equipment" does not exist`
**Causa:** `::equipment.tipo_recurso%TYPE` é sintaxe válida apenas em seções `DECLARE` do
PL/pgSQL — não em expressões SQL dentro do bloco.

**Fix:** Variável intermediária `v_enum_text text` + coerção implícita do PL/pgSQL:
```sql
-- Em DECLARE:
v_enum_text text;

-- No corpo:
SELECT e.enumlabel INTO v_enum_text
FROM pg_enum e JOIN pg_attribute a ON a.atttypid = e.enumtypid
WHERE a.attrelid = 'equipment'::regclass AND a.attname = 'tipo_recurso'
ORDER BY e.enumsortorder LIMIT 1;
v_tr := v_enum_text;  -- PL/pgSQL converte text → ENUM implicitamente em atribuição
```

#### Erro 4: `42601 INSERT has more expressions than target columns`
**Causa:** 3 INSERTs em `queue_entries` (Nilda, Quirino, Bernardo) tinham `'Montes Claros','MG'`
duplicado nos VALUES — 14 valores para 12 colunas.

**Fix:** Remoção do par duplicado nos 3 INSERTs. Mesmo padrão já corrigido em sessões
anteriores (arquivo 2 e 3 — `municipio_paciente, uf_paciente` adicionados por edição e
acidentalmente duplicados na geração).

### Lições aprendidas (2026-04-20)

#### PostgreSQL / PL/pgSQL

| Lição | Regra prática |
|-------|---------------|
| `%TYPE` em DECLARE vs em SQL | `v_tr equipment.tipo_recurso%TYPE` é válido **apenas** na seção DECLARE. Dentro do corpo do DO $$, nunca usar `::equipment.tipo_recurso%TYPE` em casts SQL — usará sintaxe regular de schema, não de tipo. |
| Coerção text → ENUM em PL/pgSQL | Atribuição `v_tr := 'algum_valor'` com `v_tr` declarado como `%TYPE` funciona automaticamente se o literal for um label válido do ENUM. Não precisa de cast explícito. |
| Leitura de ENUMs sem dados na tabela | Usar `pg_enum JOIN pg_attribute` com `attrelid = 'tabela'::regclass AND attname = 'coluna'` — retorna os labels mesmo com a tabela vazia. |
| Ordem de deleção com FKs | Ao fazer cleanup, sempre mapear todas as FKs transitivas. `appointments.equipment_id` é uma FK direta que o ON DELETE RESTRICT bloqueia mesmo quando o caminho óbvio é via `queue_entry_id`. |
| DELETE scoped vs global | **Nunca usar** `DELETE FROM tabela WHERE data_source = 'x'` como único critério quando há FKs. Sempre deletar na ordem inversa das FKs e usar critérios de escopo (CNS LIKE, CNES LIKE). |
| Contagem de colunas em INSERT | Em INSERTs longos com muitos campos, contar explicitamente colunas vs valores. Erros de duplicata de par `municipio, uf` são difíceis de ver visualmente. |

#### Estratégia de seed

| Lição | Regra prática |
|-------|---------------|
| Seed destrutivo é perigoso | Nunca usar DELETE global por `data_source` sem transação scoped que reverta se algo falhar. |
| Separar em 3 arquivos sequenciais | Infraestrutura → Pacientes/Histórico → Fila/Pipeline é a ordem correta de dependências. Permite re-executar cada bloco isoladamente ao debugar. |
| Fallback de ENUMs em seed | Todo DO $$ que cria dados com colunas ENUM deve ter fallback hardcoded (`IF v_ta IS NULL THEN v_ta := 'consulta'`) para rodar após cleanup total. |
| Validação inline com RAISE NOTICE | Adicionar `RAISE NOTICE '✅ ...'` ao final de cada sub-bloco facilita localizar onde um erro ocorreu sem re-executar tudo. |
| Verificação com SELECT ao final | Sempre fechar cada arquivo com queries de verificação em comentário — ajuda a confirmar os dados sem re-abrir o SQL Editor manualmente. |

### Lições aprendidas (2026-04-24)

#### PostgreSQL — UUIDs e lookups em seeds

| Lição | Regra prática |
|-------|---------------|
| **Nunca hardcodar UUIDs de `gen_random_uuid()`** | IDs gerados por `gen_random_uuid()` em migrations são diferentes a cada execução se a migration for re-aplicada. UUIDs obtidos de queries diagnósticas no banco atual ficam desatualizados após rollback+reseed. Use sempre `SELECT id FROM equipment WHERE nome = '...'`. |
| **Nomes de equipment usam em-dash (—)** | O padrão no banco é `'RX-01 — Aroldo Tourinho'` (U+2014), não `'RX-01 - Aroldo Tourinho'` (hífen simples). Verificar na migration de criação antes de escrever lookup. |
| **`EXTRACT(EPOCH FROM uuid_col)` não existe** | Para pseudo-aleatório baseado em UUID, usar `abs(hashtext(id::text)) % N`. |
| **`LIMIT` não funciona com `ARRAY_AGG`** | `SELECT ARRAY_AGG(id) ... LIMIT 11` aplica o LIMIT antes do ARRAY_AGG, não depois — resulta em array com até 11 elementos MAS o comportamento é dependente do plano. Usar subquery: `SELECT ARRAY_AGG(id) FROM (SELECT id ... LIMIT 11) sub`. |

#### Estratégia de seed para demo

| Lição | Regra prática |
|-------|---------------|
| **Separar em arquivos por hospital/tema** | Evita atingir limites de tamanho do SQL Editor e permite re-executar apenas o hospital quebrado. Cada arquivo deve ser 100–250 linhas, autocontido. |
| **`data_source` como identificador de rollback** | Adicionar `data_source = 'seed_final_demo'` em todos os INSERTs é a única garantia de rollback cirúrgico sem afetar dados base. |
| **Lookup de enums com fallback** | O padrão `SELECT tipo_atendimento INTO v_ta FROM queue_entries WHERE tipo_atendimento::text = 'exame' LIMIT 1; IF v_ta IS NULL THEN SELECT ... LIMIT 1; END IF;` evita crash em banco recém-limpo. |
| **Absenteísmo por fórmula modular** | `v_i % N < K` é previsível, auditável e controlado. Antes de usar, calcular `K/N` e verificar se está no range alvo. Ex: `v_i % 7 < 3` = 42,9%; `v_i % 10 < 3` = 30,0%. |
| **Intervalo de tempo = volume ÷ período** | Para distribuir N appointments em M dias: intervalo = `(M * 24) / N` horas. Verificar antes de escrever o INTERVAL para evitar todos os apts no mesmo horário ou além do período. |
| **Validar com V1-V10 antes de gravar** | Nunca gravar vídeo sem executar as queries de verificação. KPIs podem parecer corretos mas estar calculados com denominador errado. |

**Sintoma:** Cards do Board 1 (agendamentos sem confirmação) mostram `—` ou texto quase invisível onde deveria estar o nome do profissional/técnico.

**Causa raiz:**
O componente resolve `profissional_nome` pela seguinte cadeia de fallback (em `grupos` useMemo):
```

2. profissionais.find(p => p.ubs_id == equipment.ubs_id && tipo == 'medico')  ← fallback UBS
3. profissionais.find(p => p.ubs_id == equipment.ubs_id)  ← qualquer nessa UBS
4. equipment.profissional_nome   ← campo direto no equipment

```

### Lições aprendidas — Seed e constraints do banco

| Lição | Regra prática |
|-------|---------------|
| **Verificar is_nullable ANTES de alterar** | Antes de qualquer `ALTER TABLE`, confirmar `information_schema.columns.is_nullable`. Um `DROP NOT NULL` em produção pode ser irreversível se tiver dados dependentes. |
| **Tabelas standalone > ALTER TABLE** | Quando o módulo novo não tem dependência direta dos dados existentes, criar tabela nova é SEMPRE mais seguro. Evita constraint violations e rollback parcial. |
| **Seed com cobertura parcial gera UX inconsistente** | O seed cobriu apenas 5 de 15 equipment com `profissional_nome`. Os 10 restantes ficaram com `—`. Para demo, TODOS os equipment ativos devem ter profissional atribuído. |
| **`patient_id NOT NULL` bloqueia dual-insert** | A estratégia de "espelhar" uma notificação num `notification_log` para profissionais não é viável sem criar um patient fake. A extensão correta é nova aba ou nova tabela com leitura paralela na tela de WhatsApp. |
| **Rollback cirúrgico funciona via `data_source`** | O padrão `DELETE FROM tabela WHERE data_source = 'seed_xyz'` no rollback funcionou perfeitamente — apenas as linhas do seed foram removidas, sem afetar dados manuais inseridos via UI. |

### RPC rpc_kpis_profissionais — Especificação técnica

```sql
CREATE OR REPLACE FUNCTION public.rpc_kpis_profissionais(
  p_horizonte_horas integer DEFAULT 72
)
RETURNS json
LANGUAGE sql
STABLE
SECURITY DEFINER
SET search_path = public
```

**Retorno (JSON):**
```json
{
  "agendas_confirmadas_pct": 66.7,
  "equip_confirmaram": 2,
  "equip_com_agenda": 3,
  "indisponibilidades_count": 1,
  "pacientes_protegidos": 4
}
```

**KPI1 — Agendas confirmadas (%):**
- Denominador: `equipment` distintos com `appointments.status='agendado'` na janela `NOW()` a `NOW() + horizonte`
- Numerador: desses, quantos têm `professional_confirmations.status_resposta='confirmou_disponibilidade'`
- Threshold UI: verde ≥ 80%, âmbar 50–79%, vermelho < 50%

**KPI2 — Indisponibilidades (count):**
- `professional_confirmations WHERE status_resposta='reportou_indisponibilidade'`
- Janela: últimos 30 dias (`created_at >= NOW() - interval '30 days'`)

**KPI3 — Pacientes protegidos (count):**
- Subquery: slots afetados = `(equipment_id, DATE(scheduled_at))` de indisponibilidades
- Conta appointments em `status IN ('aguardando','agendado','confirmado')` nos mesmos slots
- **Bug corrigido:** `'pendente'` não existe no enum `status_local` — usava-se erroneamente, substituído por `'aguardando'` (22P02 eliminado)

### Lições aprendidas (sessão 2026-04-22)

#### Arquitetura / banco

| Lição | Regra prática |
|-------|---------------|
| `notification_log.patient_id NOT NULL` | Ao inserir notificações programáticas, nunca tentar inserir com `patient_id=null`. Navegar `appointments → queue_entries` para resolver o FK antes do INSERT. |
| `status_local` enum | Valores válidos: `aguardando`, `agendado`, `confirmado`, `realizado`, `faltou`, `cancelado`. `'pendente'` **não existe** — causava erro `22P02` nas RPCs. |
| Realtime + StrictMode | Canal Supabase deve ter nome único por instância. Usar `useRef` + `crypto.randomUUID()` para gerar sufixo — evita "already subscribed" em StrictMode (React 18). |
| Enum em migrations | Antes de usar um valor de enum em `WHERE` ou `IN`, confirmar no `pg_enum` se o label realmente existe. |

#### JavaScript / React

| Lição | Regra prática |
|-------|---------------|
| Breaking change em parâmetros | Ao mudar a assinatura de uma função utilitária (`data_longa → scheduled_at`), buscar **todos os call sites** com grep antes de commitar. |
| Toast não-bloqueante para operações secundárias | Erros em operações de "efeito colateral" (como enviar avisos preventivos) devem ser `console.warn` — nunca re-throw — para não interromper fluxo do gestor. |
| usePendingProfCount | Cada badge de nav que precisa de dados precisa de seu próprio mini-hook local no Layout; evitar prop-drilling ou contextos globais para contadores simples. |

| Lição | Regra prática |

| Commits semânticos por domínio | Separar db / frontend-novo / routing / whatsapp / assistente-ia facilita `git bisect` e revisão de PR. |
| Mensagens de commit como documentação | Commit body deve descrever *o que* e *por quê*, não só *o que*. Especialmente para migrações irreversíveis. |
