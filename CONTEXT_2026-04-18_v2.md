## 1. SISTEMA — O QUE É

**SUS Raio-X** — sistema GovTech para gestão de filas de exames de imagem e consultas
especializadas do SUS em **Montes Claros/MG**.

**Edital:** CPSI Co.NE nº 004/2026 — Prefeitura de Montes Claros / ENAP / Sudene / BID
**Submissão MVP:** 28/04/2026 | **Aceleração:** 18/mai → 10/jul/2026 | **Demoday:** 03–07/ago/2026
**Valores:** R$200k (CPSI) → R$1,6M (contrato subsequente)

**Problema resolvido:** vagas de exame canceladas em cima da hora ficavam ociosas. O sistema:
1. Convoca automaticamente o próximo paciente da fila (FIFO clínico) ao haver cancelamento
2. Monitora ocupação de equipamentos em tempo real
3. Comunica com pacientes via WhatsApp
4. Oferece assistente de IA conversacional para o gestor público

**Stack:** React 18 + Vite + Tailwind | Supabase (PostgreSQL + Realtime + Auth + RPC SECURITY DEFINER)

**Metas do edital CPSI 004/2026:**
| KPI | Baseline | Meta |
|---|---|---|
| Absenteísmo | ~35% | ≤ 15% |
| Tempo médio espera | > 120 dias | ≤ 120 dias |
| Aproveitamento de capacidade | ~65% | ≥ 85% |
| Satisfação usuário | ? | > 8/10 |

### 5.2 Configuração técnica

| Item | Valor |
|---|---|
| Modelo | `claude-sonnet-4-20250514` |
| URL API (dev) | `/api/anthropic/v1/messages` (proxy Vite) |
| URL API (prod/TODO) | Supabase Edge Function |
| Proxy Vite | `changeOrigin: true` + `proxy.on('proxyReq', removeHeader('origin'))` |
| API Key | `VITE_ANTHROPIC_API_KEY` em `.env.local` (não commitar) |

**Por que o proxy remove o header `origin`:** o Anthropic detecta requests de browser
pelo header `Origin`. Com `changeOrigin: true` o Vite troca apenas o `Host`, mas passa
adiante o `Origin`. A remoção explícita via `proxyReq.removeHeader('origin')` faz o
Anthropic tratar a request como server-to-server, eliminando o erro CORS.

#### 2.1 `get_fila_por_ubs(p_horizonte_dias, p_tipo_atendimento)`
**Responde:** "Qual UBS tem maior fila?", "Onde está a demanda represada?"
```sql
-- queue_entries com status_local IN ('aguardando','agendado') → agrupa por ubs
-- Retorno: {ubs_nome, municipio, total_aguardando, pct_do_total, espera_media_dias}
SELECT
  u.nome AS ubs_nome,
  u.municipio,
  COUNT(qe.id) AS total_aguardando,
  ROUND(COUNT(qe.id)*100.0 / NULLIF(SUM(COUNT(qe.id)) OVER (), 0), 1) AS pct_do_total,
  AVG(EXTRACT(EPOCH FROM (NOW()-qe.data_solicitacao_sisreg))/86400)::int AS espera_media_dias
FROM queue_entries qe
JOIN ubs u ON u.id = qe.ubs_id
WHERE qe.status_local IN ('aguardando','agendado')
  AND ($1 IS NULL OR qe.data_solicitacao_sisreg >= NOW() - ($1||' days')::interval)
GROUP BY u.id, u.nome, u.municipio
ORDER BY total_aguardando DESC
```
**Novo intent:** `fila_por_ubs`

#### 2.4 `get_tipos_exame_solicitados(p_horizonte_dias, p_tipo_atendimento)`
**Responde:** "Quais exames são mais solicitados?", "Tipo de procedimento mais pedido?"
```sql
-- queue_entries.nome_grupo_procedimento (com fallback para tipo_atendimento)
-- Retorno: {tipo_exame, total_solicitacoes, pct_do_total, espera_media_dias}
SELECT
  COALESCE(qe.nome_grupo_procedimento, qe.tipo_atendimento, 'Não informado') AS tipo_exame,
  COUNT(*) AS total_solicitacoes,
  ROUND(COUNT(*)*100.0 / NULLIF(SUM(COUNT(*)) OVER(), 0), 1) AS pct_do_total,
  AVG(EXTRACT(EPOCH FROM (NOW()-qe.data_solicitacao_sisreg))/86400)::int AS espera_media_dias
FROM queue_entries qe
WHERE $1 IS NULL OR qe.data_solicitacao_sisreg >= NOW() - ($1||' days')::interval
GROUP BY tipo_exame
ORDER BY total_solicitacoes DESC LIMIT 15
```
**Novo intent:** `tipos_exame`
